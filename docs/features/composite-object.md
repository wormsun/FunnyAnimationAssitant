# 组合对象（Composite Object）需求分析

> **文档状态**：需求确认 + 实现补充  
> **日期**：2026-02-28（更新）  

---

## 一、概念起源：场景与人物的同构性

### 1.1 问题背景

目前 FunnyAnimationAssistant 的对象体系呈现**两极分化**：

| 类别                       | 特征                                                    | 局限                 |
| :------------------------- | :------------------------------------------------------ | :------------------- |
| **场景对象 (SceneObject)** | 完全扁平，无内部结构，只有整体变换                      | 无法表达多部件组合   |
| **人物 (Character)**       | 完全封闭，有复杂的内部结构（Parts），但外部无法干预内部 | 黑盒设计，灵活性不足 |

两者之间缺少一种**中间形态**——既有内部结构，又允许外部干预。

### 1.2 同构性分析

通过对比发现，场景和人物在多个维度上具有高度同构性：

#### 组成方式

- **场景**：由 `SceneSetup.objects[]` 数组中的多个 `SceneObject` 组成
- **人物**：由 `Character.states[stateId].parts[]` 数组中的多个 `Part` 组成
- **共同点**：都是"容器 → 多个基本单元"的结构

#### 素材系统

- **场景对象**：`SceneObject.refId` 引用资源 ID
- **人物部位**：`Part.assets[]` 每个部位拥有自己的素材池，通过 `currentAssetId` 切换
- **共同点**：都是"一个单元 → 多个可选素材 → 运行时决定使用哪个"

#### 空间属性

| 属性   | 场景对象         | 人物部位                               |
| :----- | :--------------- | :------------------------------------- |
| 位置   | `x, y`           | `assetConfigs[id].transform.x/y`       |
| 缩放   | `scaleX, scaleY` | `assetConfigs[id].transform.scale.x/y` |
| 旋转   | `rotation`       | `assetConfigs[id].transform.rotation`  |
| 层级   | `zIndex`         | Layer Presets / `parts[]` 数组顺序     |
| 透明度 | `alpha`          | `assetConfigs[id].alpha`               |
| 可见性 | `visible`        | `assetConfigs[id].visible`             |

**结论**：如果把人物的部位视为场景对象，人物本质上就是一个"多个场景对象的组合"——这就是**组合对象**的概念来源。

## 二、组合对象的核心特性定义

### 2.1 命名约定

| 场景     | 名称                            |
| :------- | :------------------------------ |
| 用户界面 | **组合对象**                    |
| 代码内部 | `Composite` / `CompositeObject` |

选择"组合对象"而非"小场景"，原因：
- 不与"场景"混淆——场景是表演舞台，组合对象是舞台上的对象集合
- "组合"传达"多个对象拼在一起"的含义
- "对象"明确表达它是场景中的一个独立实体，可以作为整体操作

### 2.2 基本能力

组合对象是一种新增的 `SceneObject` 类型（`type: 'composite'`），具备以下能力：

| 能力              | 描述                                                             |
| :---------------- | :--------------------------------------------------------------- |
| **内部结构**      | 包含多个子对象（道具、背景片段、甚至其他组合对象）               |
| **局部坐标系**    | 子对象的位置是相对于组合对象原点的                               |
| **整体变换**      | 组合对象整体可以移动、缩放、旋转，所有子对象自动跟随             |
| **独立操作**      | 每个子对象拥有全局 ID，可被独立 Action 控制                      |
| **动态归属变更**  | 对象可通过 Action 动态加入或离开组合对象（修改 `parentId` 归属） |
| **层级调整**      | 子对象的 `zIndex` 可自由调整，实现层级穿插                       |
| **自由组合/拆分** | 可以将独立对象组合为组合对象，也可以拆分为独立对象               |
| **组合模式**      | 分为「所有」和「绑定」两种模式，决定消亡/删除时的子对象处理策略  |

### 2.3 使用场景示例

| 示例     | 组成                                        |
| :------- | :------------------------------------------ |
| 汽车     | 车身 + 车轮 × 4 + 车门 + 车窗               |
| 椅子     | 椅子腿 × 4 + 椅子座 + 靠背                  |
| 餐桌组合 | 桌子 + 椅子A + 椅子B + 餐具                 |
| 卧室     | 床（床头板+床垫+被子+枕头） + 床头柜 + 衣柜 |

## 三、核心议题的分析结论

### 3.1 子对象存储方案：平铺 + parentId

子对象与父对象**平铺存储**在同一个 `setup.objects[]` 数组中，通过 `parentId` 和 `childIds` 建立引用关系：

```typescript
setup.objects = [
  { id: 'composite_1', type: 'composite', childIds: ['part_1', 'part_2'], ... },
  { id: 'part_1', type: 'character_part', parentId: 'composite_1', ... },
  { id: 'part_2', type: 'expression_part', parentId: 'composite_1', ... },
  { id: 'bg_1', type: 'background', ... }  // 独立对象，无 parentId
]
```

**选择平铺方案的理由**：

| 维度               |                 平铺 + parentId                  |
| ------------------ | :----------------------------------------------: |
| Action target 查找 | ✅ O(1) 直接 `objects.find(o => o.id === target)` |
| 与现有代码兼容     |            ✅ 不需修改任何现有查找逻辑            |
| Vue 响应式         |               ✅ 扁平数组天然响应式               |
| 序列化             |             ✅ 无嵌套，JSON 直接平铺              |
| 删除组合对象       |      需同步删除所有 `parentId` 匹配的子对象      |

### 3.2 坐标系：局部坐标 + PIXI 容器嵌套

- 子对象的 `x/y` 是**相对于父对象原点**的局部坐标
- 渲染时，子对象 PIXI 容器嵌套在父对象容器内，PIXI 自动处理坐标级联
- 移动组合对象 = 仅修改组合对象的 `x/y`，子对象坐标不变
- 组合时：全局坐标 → 局部坐标（PIXI `toLocal()`）
- 拆分时：局部坐标 → 全局坐标（PIXI `toGlobal()`）

#### 前置条件：坐标语义统一（v2.0.0，实现中新增）

实现组合对象嵌套容器时发现，系统中存在**两种坐标语义并存**的问题：

| 对象类型     | 原 x/y 语义                      | 统一后 |
| :----------- | :------------------------------- | :----- |
| Camera       | 中心坐标                         | 中心   |
| Character    | 混合（创建时中心，渲染时左上角） | 中心   |
| Prop         | 左上角                           | 中心   |
| Background   | 左上角 (0,0)                     | 中心   |
| ScreenEffect | 中心坐标                         | 中心   |

**解决方案**：统一所有对象类型的 `x/y` 语义为**中心坐标**，消除 `applyTransform` 中的 offset 补偿逻辑。这是 **Breaking Change**（v1.0.0 → v2.0.0），需要自动迁移机制。

### 3.3 层级穿插 — 通过 zIndex 自由调整

**需求**：汽车内部的座位上可以放另一个对象（人物、包等），该对象需要渲染在组合对象特定部件的层级之间。

**方案**：每个子对象拥有独立的 `zIndex`，可自由调整层级顺序。外部对象也可以添加为组合对象的子对象，通过设置合适的 `zIndex` 实现层级穿插。

- 不需要预定义"挂载点"或"插槽"概念
- 用户操作：拖一个道具进组合对象，调整层级顺序即可
- 子对象的 `zIndex` 仅在组合对象内部生效（局部层级）

#### 同 zIndex 子对象的渲染排序（实现中新增）

当多个子对象拥有相同 `zIndex` 时，需要额外的排序机制：

- **`childIds` 数组顺序决定渲染先后**：数组前面的先渲染（在底层）
- **渲染层实现**：通过 `container.zIndex` 添加 `childIndex × 0.001` 微偏移，使 PIXI `sortableChildren` 按 `childIds` 顺序排列
- **编辑器 UI**：属性面板中按 `zIndex` 分组显示子对象，同组内支持拖拽排序（HTML5 DnD）
- **数据层**：Store 提供 `reorderChild(compositeId, fromIndex, toIndex)` 方法

### 3.4 临时组合 — 运行时动态成组

**需求**：人物拉行李箱一起移动，之后又分开。

**方案**：不需要新增 `group`/`ungroup` Action，全部通过现有 Action 组合实现：

**成组流程**（编辑器 UI 层封装为“成组”按钮，内部拆解为多个 Action）：
1. Setup 中预定义一个 `CompositeObject`（`spawned: false`）
2. `set_lifecycle` — 激活组合对象
3. `set_parent` × N — 将各对象加入组合（坐标自动从全局转换为局部）

**拆分流程**：
1. `set_parent`（parentId: null）× N — 将各对象移出（坐标自动从局部还原为全局）
2. `set_lifecycle` — 隐藏空壳组合对象

**设计原则**：不引入“跟随”等中间态概念，保持用户心智模型的简洁——**要么组合在一起，要么各自独立**。

> **坐标补偿规则**：`set_parent` Action handler 在执行时**自动完成完整矩阵变换坐标转换**——沿 parentId 链递归计算世界变换矩阵，通过矩阵求逆将子对象从旧坐标系映射到新坐标系（x/y/scaleX/scaleY/rotation 全量补偿），无需额外生成 `set_transform` Action。这保证了画面不会因为归属变更而发生视觉跳跃。

### 3.5 自由组合与拆分

**核心挑战**：坐标系转换。

- **组合时**：全局坐标 → 局部坐标（相对于组合对象原点）
- **拆分时**：局部坐标 → 全局坐标（还原到世界坐标系）
- 如果组合对象有旋转和缩放，需要矩阵运算（PIXI 的 `toLocal()` / `toGlobal()` 天然支持）

**设计原则**：
- 拆分后恢复为独立场景对象，保留原始 `id`，确保 Action 引用不断裂
- 组合后的 Action 可以 target 整个组合对象（整体操作），也可以 target 内部成员（单独操作）

### 3.6 组合对象的内部动画（关键设计决策）

**核心矛盾**：FunnyAnimationAssistant 的时间是由剧本驱动的（文本 → TTS → Slot → Action），组合对象内部没有剧本，如何驱动内部动画？

**结论：组合对象内部不需要 Action 系统，也不需要剧本驱动。**

组合对象需要的是**预定义的状态和动画**，由外部场景的 Action 来触发。

#### 三种方案的对比

| 方案                          | 描述                                             | 复杂度 | 用户理解 | 推荐 |
| :---------------------------- | :----------------------------------------------- | :----- | :------- | :--- |
| A. 内部 Action 系统           | 组合对象内有自己的 Slot/Action/时间线            | 🔴 极高 | 🔴 很难   | ❌    |
| B. 资源级 AnimationDefinition | 复用现有动画系统，通过 `set_anim` 从外部触发     | 🟡 中等 | 🟡 一般   | ✅    |
| C. 纯状态切换                 | 组合对象有离散预设状态，通过 Action 切换（硬切） | 🟢 低   | 🟢 很容易 | ✅    |

#### 推荐方案：C + B 结合，分阶段实现

**第一阶段（C — 状态切换）**：
- 组合对象定义多个预设状态（类比人物姿态）
- 每个状态定义子对象的 PRS + 可见性 + 素材选择
- 在场景中通过 Action 切换状态（硬切）
- 用户心智模型："这就像人物换姿态一样"

**第二阶段（B — 资源级动画）**：
- 组合对象定义 `AnimationDefinition`（关键帧驱动子对象变换）
- 复用现有 `set_anim` Action 从场景中触发
- 动画时间是自包含的（AnimationDefinition 自带 duration），不依赖场景 Slot

**时间驱动关系**：
```
场景剧本（提供宏观时间线）
  └── Slot N: set_anim { target: "汽车", animName: "行驶" }
        └── 汽车内部动画（自包含时间循环，不依赖外部 Slot）
```

> 场景负责"何时触发"，组合对象负责"触发后做什么"。

### 3.7 嵌套组合

**需求**：组合对象可以包含组合对象。

**结论：必须支持，不做硬性层级限制。**

示例：
```
餐厅 (组合对象)
  ├── 桌子 (原子道具)
  ├── 椅子A (组合对象)
  │     ├── 椅子腿 (原子道具)
  │     ├── 椅子座 (原子道具)
  │     └── 靠背 (原子道具)
  ├── 椅子B (组合对象，同结构)
  └── 人物 (角色对象，坐在椅子A上)
```

在平铺存储下的表示：
```typescript
setup.objects = [
  { id: 'restaurant', type: 'composite', childIds: ['table', 'chairA', 'chairB', 'person'] },
  { id: 'table', type: 'prop', parentId: 'restaurant' },
  { id: 'chairA', type: 'composite', parentId: 'restaurant', childIds: ['leg1', 'seat', 'back'] },
  { id: 'leg1', type: 'prop', parentId: 'chairA' },
  { id: 'seat', type: 'prop', parentId: 'chairA' },
  { id: 'back', type: 'prop', parentId: 'chairA' },
  { id: 'chairB', type: 'composite', parentId: 'restaurant', childIds: [...] },
  { id: 'person', type: 'character', parentId: 'restaurant' }
]
```

**保护措施**：
- 禁止循环引用（A 包含 B，B 包含 A），加检测
- 深层嵌套（超过 3-4 层）时给出性能提示
- 编辑器 UI 使用树形面板，支持展开/折叠，支持"双击进入 / Esc退出"层级编辑

## 四、与现有架构的兼容性原则

### 4.1 增量式演进，不做大重构

| ❌ 不应该做                     | ✅ 应该做                                                  |
| :----------------------------- | :-------------------------------------------------------- |
| "万物皆 Node"的大重构          | 组合对象作为新的 `type` 加入现有 `SceneObject` 体系       |
| 序列化格式从扁平数组改为递归树 | 子对象平铺存储，通过 `parentId` / `childIds` 建立引用关系 |
| Action 系统全面改为路径寻址    | 保持扁平寻址，子对象拥有全局 ID，可被直接 target          |
| 人物系统重构为组合对象子类     | 并行演进，共享底层能力                                    |

### 4.2 时间模型不变

- **Action Mode 下严禁使用具体时间**（核心规则不变）
- 组合/拆分等操作通过 Action 在 Slot 中触发
- 组合对象内部动画的时间是**自包含的**，不依赖外部 Slot

### 4.3 对象身份稳定性

- 对象加入或离开组合对象时，保留原始 `id`
- 确保已有的 Action 引用（`target`）不会断裂

### 4.4 Action 控制方式

部件作为独立场景对象后，现有 Action 全部直接可用。新增两种 composite 相关 Action：
- `set_parent`：修改子对象的归属关系（target = 子对象）
- `set_composite`：修改 composite 自身属性（target = composite 对象），如 `compositeMode`、`childIds` 排序等

```typescript
// 移动某个子对象
{ type: "set_transform", target: "part_expr_001", params: { x: 120, y: -60 } }

// 子对象渐变动画
{ type: "tween_transform", target: "part_body_001", params: { scaleX: 1.2 }, duration: 500 }

// 将一个对象动态加入组合对象（修改归属关系）
{ type: "set_parent", target: "sword_001", params: { parentId: "composite_char_001" } }

// 将一个对象从组合对象中移出（恢复为顶层独立对象）
{ type: "set_parent", target: "sword_001", params: { parentId: null } }

// 修改子对象的渲染排列顺序
{ type: "set_composite", target: "composite_001", params: { childIds: ["B", "A", "C"] } }

// 切换组合模式
{ type: "set_composite", target: "composite_001", params: { compositeMode: "own" } }

// 切换子对象资源
{ type: "set_asset", target: "part_hand_001", params: { refId: "fist_asset" } }
```

#### `set_parent` vs `set_composite` 职责划分（实现中确认）

| Action          | target         | 修改的字段                      | 说明                       |
| :-------------- | :------------- | :------------------------------ | :------------------------- |
| `set_parent`    | 子对象         | `parentId`                      | 修改归属关系，自动坐标补偿 |
| `set_composite` | composite 对象 | `compositeMode` / `childIds` 等 | 修改 composite 自身属性    |

`set_composite` 遵循现有 Action 体系的"字段族合一"设计模式（类比 `set_visual` 合并 visible/flipX/zIndex），将 composite 特有属性统一到一个 Action 类型中。后续新增 composite 属性时只需扩展 `SetCompositeParams`，无需创建新的 Action 类型。

## 六、术语对照表

| 用户界面术语 | 代码术语                        | 描述                                                     |
| :----------- | :------------------------------ | :------------------------------------------------------- |
| 组合对象     | `Composite` / `CompositeObject` | 多个场景对象的组合                                       |

## 七、复杂度分析

### 组合对象 vs 人物部位外部控制

**结论：人物部位外部控制更复杂。**

| 维度             | 组合对象                               | 人物部位外部控制                                          |
| :--------------- | :------------------------------------- | :-------------------------------------------------------- |
| **数据模型**     | 🟢 新建 `type: 'composite'`，无历史包袱 | 🔴 在 Character/Part/AssetConfig 上叠加，需兼容已有数据    |
| **渲染器改动**   | 🟢 复用 `SceneObjectRenderer` 与 Provider 注册体系 | 🔴 需要改动旧式人物专用渲染链并兼容历史数据 |
| **Action 系统**  | 🟢 复用现有 Action，无需新增类型        | 🔴 需与 `set_character`、Virtual Bones、Layer Presets 协调 |
| **坐标系**       | 🟢 简单的 Container 父子关系            | 🔴 受 AssetConfig、Virtual Bones、Center-Mapped 多重影响   |
| **跨引擎一致性** | 🟡 ScenePlayer + FrameCapture 需同步    | 🔴 三大引擎的人物渲染逻辑都要改                            |
| **测试风险**     | 🟢 新功能，从零写测试                   | 🔴 需回归测试所有现有人物功能                              |

## 八、编辑器 UI 需求

### 8.1 对象选择器：树形显示

Setup Mode 和 Action Mode 的属性面板中，对象选择器从扁平 `<select>` 改为**树形下拉列表**：

```
选中对象: [📦 汽车 🔒 ▾]           ← 当前选中项 + 点击展开下拉
┌──────────────────────────────┐
│ 📷 相机                      │
│ 🖼️ 背景1                    │
│ 📦 汽车 🔒                 ▼ │  ← composite 节点，可展开/折叠
│   ├ 🎨 车身                  │  ← 子对象缩进显示
│   ├ 🎨 车轮1                 │
│   └ 🎨 车轮2                 │
│ 👤 人物1                     │
│ 📦 餐桌 🔓                 ▶ │  ← 折叠态，▶ 表示可展开
└──────────────────────────────┘
```

**交互规则**：

| 操作           | 行为                                     |
| :------------- | :--------------------------------------- |
| 点击任意对象行 | 选中该对象（包括子对象）                 |
| 点击 ▶/▼ 图标  | 展开/折叠 composite 子列表（不改变选中） |
| 点击 🔒/🔓 图标  | 切换 composite 的 `compositeLocked` 状态 |

### 8.2 组合对象锁定/解锁模式

组合对象拥有 `compositeLocked` 属性，控制画布上的交互行为。

> **UI-Only 属性**：`compositeLocked` 是纯编辑器 UI 辅助属性，**不保存到项目文件中**，默认值为 `true`（锁定）。在 Setup Mode 和 Action Mode 下均可直接切换，无需创建 Action。

| 状态               | 画布交互                                    | 适用场景            |
| :----------------- | :------------------------------------------ | :------------------ |
| **🔒 锁定**（默认） | 点击子对象 → 选中 composite 整体 → 整体拖动 | 移动/缩放整个组合体 |
| **🔓 解锁**         | 点击子对象 → 直接选中子对象 → 独立操作      | 编辑内部布局        |

**画布 hit-test 逻辑**：

```
用户点击画布 → PIXI hit-test → 命中容器 X
  → X 有 parentId?
    → 查找父级 composite
      → compositeLocked === true → 选中 composite（整体操作）
      → compositeLocked === false → 选中 X（独立操作）
  → X 无 parentId → 选中 X（正常行为）
```

**切换入口**：
- 对象选择器树中 composite 行的 🔒/🔓 按钮
- 属性面板中 composite 的基础属性区

### 8.3 成组模式（Grouping Mode）

不依赖画布多选机制，通过进入**临时成组模式**实现多对象快速成组。

#### 入口

工具栏新增 **[📦 成组]** 按钮，点击后进入成组模式。

#### 交互流程

```
工具栏: [+ 添加素材] [❐ 复制] [🗑️ 删除] [📦 成组]

用户点击 [📦 成组] →
┌───────────────────────────────────────────────────────────────┐
│ 📦 成组模式 — 点击画布上的对象加入组合                         │
│ 已选: 👤人物1, 📦行李箱                     [✓ 完成] [✗ 取消] │
└───────────────────────────────────────────────────────────────┘

• 每点击一个对象 → 加入待成组列表（对象高亮标记）
• 再次点击已选对象 → 从列表中移除（取消高亮）
• 点击 [✓ 完成] → 调用 groupObjects(pendingIds)，创建 composite
• 点击 [✗ 取消] 或 ESC → 退出模式，不做任何操作
```

### 8.4 添加成员模式（Add-to-Composite Mode）

选中已有 composite 时，可快速将画布上的独立对象加入此组合。与成组模式共享同一套 UI 和状态机。

#### 入口

### 8.9 Action Mode：Sequencer 中的组合对象

#### 8.9.1 轨道分组显示

Sequencer 中 composite 轨道作为**父轨道**，子对象轨道缩进显示为**子轨道**，支持折叠/展开：

```
📷 相机        │ ✂️━━━━  🎬━━━━━━━━  │
📦 汽车      ▼ │ ━━━(整体移动)━━━━━  │  ← composite 父轨道（可折叠）
  🎨 车身      │                     │  ← 子轨道，缩进显示
  🎨 车轮1     │ ━(旋转)━━           │
  🎨 车轮2     │ ━(旋转)━━           │
👤 人物1       │ ━━━(walk)━━━        │  ← 独立轨道
```

折叠态：只显示 composite 父轨道，子轨道隐藏。

#### 8.9.2 轨道分组依据：Block 累积状态 + 当前 Block Action

轨道分组**按 block 动态计算**，依据为：

```
轨道分组 = Block 初始状态中的 parentId + 当前 Block 内所有 set_parent Action 的效果
```

- Block 初始状态 = Setup 中的 parentId/childIds + 前序所有 Block 的 set_parent 累积
- 当前 Block 中添加 `set_parent` Action 后，Sequencer **立即重新分组**

**示例**：

**Block 1**（Setup 中人物和行李箱各自独立）：
```
👤 人物1     │ ━(walk)━━━          │  ← 独立轨道
📦 行李箱    │       set_parent ─→ │  ← 独立轨道，添加 set_parent 后立即重排 ↓
```
添加 `set_parent { target: "行李箱", parentId: "人物组合" }` 后，轨道立即变为：
```
📦 人物组合  │                     │  ← 新出现的 composite 父轨道
  👤 人物1   │ ━(walk)━━━          │  ← 子轨道
  📦 行李箱  │       set_parent ─→ │  ← 子轨道
```

**Block 2**（Block 1 的组合关系已累积）：
```
📦 人物组合  │ ━━(整体移动)━━━━━   │
  👤 人物1   │                     │
  📦 行李箱  │                     │
```

**Block 3**（Block 2 已执行 set_parent(null) 拆分）：
```
👤 人物1     │ ━━━(run)━━━         │  ← 恢复为独立轨道
📦 行李箱    │                     │  ← 恢复为独立轨道
```

#### 8.9.3 组合对象 Action 与子对象 Action 的关系

**完全独立，互不影响**。级联效果由 PIXI 容器层级自动处理，无需"级联 Action"概念。

| 操作       | target    | 效果                                           |
| :--------- | :-------- | :--------------------------------------------- |
| 整体移动   | composite | 移动 composite 的 x/y，PIXI 自动级联所有子对象 |
| 子对象移动 | 子对象 ID | 仅修改子对象在**局部坐标系**内的偏移           |
| 整体缩放   | composite | 缩放整个容器，子对象跟随                       |
| 子对象旋转 | 子对象 ID | 子对象独立旋转                                 |

#### 8.9.4 对象选择器

Action Mode 的对象选择器同样使用 8.1 的树形显示，分组依据与 Sequencer 一致。

### 8.10 组合模式：所有 vs 绑定

#### 8.10.1 概念

组合对象拥有 `compositeMode` 属性，区分两种归属语义：

| 模式               | 代码值   | 适用场景                   | 消亡/删除时子对象行为          |
| :----------------- | :------- | :------------------------- | :----------------------------- |
| **📦 所有**（默认） | `'own'`  | 汽车拥有车轮、椅子拥有椅腿 | 子对象**级联**消亡/删除        |
| **📎 绑定**         | `'bind'` | 人物 + 行李箱临时同行      | 子对象**冒泡**，恢复为独立对象 |

#### 8.10.2 数据模型

```typescript
interface CompositeObject extends SceneObject {
  type: 'composite'
  childIds: string[]
  compositeLocked: boolean  // UI-only，不保存到项目文件
  compositeMode: 'own' | 'bind'
}
```

#### 8.10.3 生命周期行为对照

| 事件                  | `own` 模式                     | `bind` 模式                          |
| :-------------------- | :----------------------------- | :----------------------------------- |
| 删除 composite        | 子对象**级联删除**             | 子对象冒泡（parentId → 上级或 null） |
| 消亡（spawned→false） | 子对象**级联消亡**             | 子对象冒泡                           |
| 手动拆分              | 子对象冒泡（两种模式一致）     | 子对象冒泡（两种模式一致）           |
| 复制 composite        | 递归复制子对象（两种模式一致） | 递归复制子对象（两种模式一致）       |

**冒泡规则**：子对象的 `parentId` 设为 composite 的 `parentId`（即上一级），而非直接变为 `null`。嵌套场景下孙子对象回到爷爷 composite。

#### 8.10.4 UI 体现

属性面板中 composite 的基础属性区增加组合模式切换：

```
组合模式: [📎 绑定] [📦 所有]
```

#### 8.10.5 创建时的默认值

| 创建入口             | 默认 compositeMode | 理由                   |
| :------------------- | :----------------- | :--------------------- |
| 工具栏 [📦 成组] 按钮 | `bind`             | 用户意图通常是临时绑定 |
